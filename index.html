Web Workers API

Web Workers provide a simple means for web content to run scripts in background threads.

Once created, a worker can send messages to the spawning task by posting messages to an event handler specified by the creator.

Why Web Workers?

JavaScript codebases continue to grow.

As more JavaScript runs, the more applications block ... waiting for code to finish.

How do we create and keep our existing applications responsive and high performing?

And, how do we design HTML5 applications frameworks for the future?

And, how do we prevent this?

How are events handled in the browser?

So, what can create unresponsiveness in the browser?

  • Processing large arrays or JSON responses
  • Artificial Intelligence Libraries
  • Prefetching and/or caching data for later use
  • Real-time text formatting
  • Spell checker
  • Analyzing video or audio data
  • Background I/O or polling of webservices
  • Image filtering in canvas
  • Updating many rows of a local web database

So, these are good candidates for Web Workers.

What are Web Sockets?

Full-duplex, bi-directional communication over the Web

Both the server and client can send data at any time, or even at the same time.

Enabling Real Time UI Updates without a page refresh.

This Creates Opportunity for Utopian User Experiences

Only the data itself is sent, without the overhead of HTTP headers and cookies, dramatically reducing bandwidth.

So isn't this called Comet? ... and don't these apps exist today.

Before Web Sockets, developers had 2 choices:

  • Sequentially poll the server at a regular interval.
  • Keep the connection open.

These techniques work, but it is not ideal.

And there are major limitations.

After all, HTTP was not designed for Real Time data.
